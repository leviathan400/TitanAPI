# Layer1 - a Layer-1-only Outpost 2 test mission

A small, complete Outpost 2 mission DLL built on **Layer 1 only** (`op2::abi`, our clean-room
reverse-engineering substrate) - **no facade**. It's the proof that the whole pipeline works end to end:

> **RE facts → generated `op2::abi` headers → `memory.hpp` reloc/thunks → live `Outpost2.exe`.**

It builds to `cLayer1.dll`, loads as a Colony game called **"Layer1 Test Mission"**, and stands up a small
working **Plymouth** colony - entirely by calling straight into the running game.

**Verified in-game** on OPU 1.4.1 (MSVC 19.51 / VS2026, x86, `/std:c++23preview`, static CRT; the DLL
imports only `KERNEL32`, ~86 KB). In-game it shows Power 450, 5000 common ore in active storage, and a
40-colonist population - see `D:\Outpost 2\OPU\logs\cLayer1.log`.

## What the mission does

`InitProc` (in `src/mission.cpp`):

1. **Sets up player 0** by writing the engine's `PlayerImpl` directly:
   - faction **Plymouth** (`isEden_ = 0`) + **human** (`GoHuman()`),
   - **population** 20 workers / 10 scientists / 10 kids,
   - food 5000 (cap 10000), common ore 5000 (cap 10000),
   - **tech level 12** (`Research::SetTechLevel`).
2. **Places 7 units** for player 0: Command Center, **MHD Generator**, **Common Ore Storage**, a ConVec,
   two Microwave Lynx, and a Scout.

Everything is logged step-by-step (open/close per line, so the last line survives even a crash).

## What it proves - the full Layer-1 toolkit, live

Every binding primitive in `op2::abi` is exercised against the real game here:

| Primitive | Used for | Address |
|---|---|---|
| free `__fastcall` thunk (`op2::abi::callFast`) | `Game::CreateUnit` (struct-by-value arg) | `0x478780` |
| `__thiscall` member thunks (`op2::abi::member`) | `PlayerImpl::GoHuman`, `Research::SetTechLevel` | `0x4906C0`, `0x473030` |
| global/data read (`op2::abi::ref`) | player-array base (a hardcoded immediate in OP2 code) | `0x4890C3` |
| direct struct-field writes | `PlayerImpl` fields (offsets verified by `offsetof`, `sizeof==3108`) | - |

## Build

Requires Visual Studio 2026 (MSVC; 2022+ also works) with the C++ **x86** toolset. The 32-bit platform is
mandatory - Outpost 2 is a 32-bit process (CMake hard-errors otherwise).

```
cmake -S . -B build -G "Visual Studio 18 2026" -A Win32
cmake --build build --config Release
```
→ `build/Release/cLayer1.dll`

## Deploy & run

`MapName` / `TechtreeName` in `mission.cpp` (`on6_01.map` + `MULTITEK.TXT`) must exist in your install -
they're the stock Colony combo the OP2DotNet SDK uses, so they're already in your OPU.

```
copy /Y build\Release\cLayer1.dll  "D:\Outpost 2\OPU\maps\"
```
Launch via the OPU launcher → **Colony Games** → **"Layer1 Test Mission"**. (For the 1.3.6 slim dev install,
copy it into the game root and start it via your `OP2SdkLauncher` flow.)

## Coordinates

Mission code uses **in-game (visible) tile coordinates** - what you see on screen. Outpost 2 keeps an
off-screen map border, so `eng::CreateUnit` adds a pad before talking to the engine:

```
internal = visible + (kPadX, kPadY)     // kPadX = 32, kPadY = 0   (observed on this build)
```
So `eng::CreateUnit(MapID::CommonStorage, 51, 78, 0)` places it exactly where you see tile `51,78`. If a
placement ever lands off by one, the two constants in `mission.cpp` are the single knob to adjust.

## Logging

The DLL writes `<Outpost2.exe dir>\OPU\logs\cLayer1.log` (raw KERNEL32, no extra deps). It records DLL
attach + the resolved log path, every player-setup step, and each `CreateUnit` before/after with the
returned unit id, then the `AIProc` first tick and detach. The first log line prints the exact path it used.

## Notes

- **Plymouth/Eden:** the units used can be placed for both factions, so switching `isEden_` doesn't break anything.
- **Why ore persists but needs storage:** OP2 derives metals (common/rare ore) capacity from buildings, so
  `commonOre_` only "sticks" because the Common Ore Storage is present and active. Food and population are
  raw pools and persist regardless.
- **Morale is low** (40 colonists, no housing). That's fine for a basic test; add a Nursery + Residence to
  recover it.
- **Addresses are for the OP2 1.x executable** (the community RE base) - verified on 1.3.6 and 1.4.1.
